# CRM Система учета обращений клиентов

## Описание проекта

CRM-модуль — внутреннее веб-приложение банка для централизованного управления обращениями клиентов. Система обеспечивает:

- ✅ Регистрацию обращений с указанием канала поступления
- ✅ Распределение обращений между операторами
- ✅ Отслеживание статусов обработки
- ✅ Ведение истории переписки
- ✅ Прикрепление файлов
- ✅ Контроль сроков обработки (SLA)
- ✅ Автоматический аудит изменений
- ✅ JWT-аутентификация с разграничением ролей (Оператор, Старший оператор, Администратор)

## Архитектура

Проект построен по принципам Clean Architecture и разделён на 4 независимых модуля:

### 1. **CRM.Domain**
- Доменные сущности (Ticket, Client, Operator, Message, Attachment, Status, Priority, Channel, TicketHistory)
- Не зависит от других модулей
- Содержит бизнес-логику

### 2. **CRM.Application**
- Сервисы, реализующие бизнес-логику
- DTO для обмена данными
- Интерфейсы сервисов
- Зависит от Domain

### 3. **CRM.Infrastructure**
- Работа с базой данных (ApplicationDbContext)
- Entity Framework Core миграции
- Реализация репозиториев
- Зависит от Application и Domain

### 4. **CRM.API**
- REST контроллеры
- Swagger/OpenAPI конфигурация
- Dependency Injection настройка
- Точка входа приложения (Program.cs)

## Установка и запуск

### Предварительные требования

- .NET 8 SDK
- PostgreSQL 12+
- Git

### Шаги установки

1. **Клонируйте репозиторий**
```bash
git clone https://github.com/bank/CRM.git
cd CRM
```

2. **Установите зависимости**
```bash
dotnet restore
```

3. **Настройте базу данных**

Отредактируйте `appsettings.json`:
```json
"ConnectionStrings": {
  "DefaultConnection": "Host=localhost;Port=5432;Database=CRMDb;Username=postgres;Password=ваш_пароль"
}
```

Убедитесь, что PostgreSQL запущен:
```bash
# Windows
pg_ctl start

# Linux/Mac
brew services start postgresql
```

4. **Запустите приложение**
```bash
dotnet run
```

Приложение запустится на:
- HTTPS: https://localhost:7098
- HTTP: http://localhost:5098

5. **Инициализируйте систему**

Сделайте запрос для создания администратора:
```bash
POST https://localhost:7098/api/init/setup
```

Ответ:
```json

    "id": 1,

    "firstName": "Admin",
```

## Ролевая модель доступа

Система поддерживает три роли с разными правами:

### 1. **Оператор** (Operator)
- Просмотр обращений
- Создание обращений
- Добавление сообщений
- Изменение статусов своих обращений

### 2. **Старший оператор** (SeniorOperator)
- Все права оператора
- Управление всеми обращениями
- Распределение обращений между операторами
- Создание/редактирование клиентов и операторов
- Просмотр аудита

### 3. **Администратор** (Admin)
- Полный доступ к системе
- Управление пользователями и ролями
- Настройка статусов, приоритетов, каналов
- Управление всеми аспектами системы

## Аудит изменений

Все изменения обращений автоматически логируются в таблицу `TicketHistories`. Для каждого изменения фиксируются:

- ID обращения
- Название изменённого поля
- Старое значение
- Новое значение
- Пользователь, выполнивший изменение
- Дата и время операции

### Пример записи аудита

```json
{
  "id": 1,
  "ticketId": 5,
  "fieldName": "Status",
  "oldValue": "New",
  "newValue": "InProgress",
  "changedBy": "operator1",
  "changedAt": "2026-07-06T10:30:00Z"
}
```

## Контроль SLA

Система поддерживает контроль сроков обработки обращений:

- Каждому приоритету соответствует определённый SLA
- Система автоматически отслеживает время обработки
- История изменений статусов позволяет рассчитывать метрики производительности

## Решение проблем

### Ошибка подключения к БД

Убедитесь, что:
- PostgreSQL запущен
- Строка подключения верна в appsettings.json
- Пользователь `postgres` имеет права доступа
- База данных `CRMDb` создана или будет создана автоматически

### JWT Token истёк

Сделайте повторный вход через `/api/operators/login` для получения нового токена.

### Миграция БД не применилась

При первом запуске приложение автоматически применяет миграции. Если возникли проблемы:

```bash
dotnet ef database update
```
